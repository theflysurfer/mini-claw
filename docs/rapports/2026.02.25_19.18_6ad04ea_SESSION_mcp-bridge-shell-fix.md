## Decisions
1. **Top-level `await` for tool registration**
   - **Choix**: Use top-level `await` in the extension's `export default function()` to discover and register MCP tools synchronously at load time
   - **Rationale**: `pi.registerTool()` must be called during the synchronous extension body execution â€” not in async event handlers â€” otherwise tools aren't included in the system prompt
   - **Alternative rejetee**: Lazy registration on first use (wouldn't appear in system prompt)

2. **`shell: true` for spawn on Windows**
   - **Choix**: Add `shell: true` to all `spawn("pi", ...)` calls
   - **Rationale**: On Windows, `pi` is an npm global binary resolved via PATH through the shell; without `shell: true`, `spawn` cannot find the executable
   - **Alternative rejetee**: Using absolute path to pi binary (fragile, non-portable)

3. **SSE response parsing**
   - **Choix**: Parse `text/event-stream` `data:` lines to extract JSON-RPC responses from MetaMCP
   - **Rationale**: MetaMCP returns SSE format by default; parsing data lines is the reliable approach
   - **Alternative rejetee**: Requesting a different content type (not supported by MetaMCP)

## Problems
1. **Pi spawn failing on Windows**
   - **Symptome**: `spawn` calls to `pi` binary failing silently or with ENOENT
   - **Cause**: Windows requires `shell: true` for PATH resolution of npm global binaries
   - **Solution**: Added `shell: true` to all 3 spawn sites in `pi-runner.ts`
   - **Prevention**: Always use `shell: true` on Windows when spawning PATH-resolved commands
